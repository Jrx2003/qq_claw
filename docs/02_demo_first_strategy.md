# 02 Demo-first 策略

## 为什么必须 Demo-first
这不是生产可用性评测，而是产品比赛。评委的核心工作不是深度探索系统边界，而是快速判断：
- 你找的问题是否真实
- 你的 AI 是否在核心环节不可替代
- 你的方案是否像一个真正会被用户使用的产品
- 你的表达是否清楚、完整、可信

因此，P0 版本必须优先保证：
- 逻辑被理解
- 价值被感知
- 演示稳定
- 视觉可信
- 交互可控

## 核心原则
### 1）评委不需要“自由探索”
自由探索会带来两个问题：
- 很容易走偏，体验不到核心价值
- 很容易触发未覆盖分支，导致 Demo 不稳定

因此我们采用：
- 引导式选项
- 受控按钮
- 剧情化推进
- 预置 NPC 行为

### 2）评委只扮演一个用户
评委只控制：
- 当前群里的一位真实用户（默认发起者）

其他角色：
- NPC 用户 1 / 2 / 3 自动发言
- 虾局长自动响应
- 系统自动更新时间线与卡片状态

### 3）系统是“会演示的产品”，不是“等待输入的聊天框”
交互重心放在：
- 成局卡
- 投票卡
- 确认卡
- 回忆卡
- 选项 chips
- 快捷按钮

文本输入框可以保留 UI 形态，但默认：
- 不作为主要输入方式
- 或仅接受少量预置指令
- 或点击快捷建议后自动填充

### 4）必须支持两个模式
#### 互动模式
给评委现场点击，体验产品逻辑。

#### 自动播放模式
给录屏使用，一键跑完整条故事线。

最好再有：
#### 开发模式
可跳场景、重播步骤、查看状态。

## 演示逻辑
### 主线只保留一个场景
周五宿舍烤肉局

这是最容易理解的场景：
- 高频
- 低解释成本
- 容易产生“我都可以”的决策停滞
- 能自然展示投票、提醒、回忆

### 扩展场景只做成“可切换展示”
- 匿名倡议
- 冲突中继
- 腾讯游戏局

不进入主流程，不抢主闭环叙事。

## Demo 中最该被放大的点
### A. 不想当组织者
让虾局长“代替人出头”，承担收口和推进责任。

### B. 匿名表达
让不想暴露自己的人，也能安全提出想法。

### C. “我都可以”治理
用 AI 发起有效投票，而不是让群继续耗散。

### D. 回忆沉淀
这是和普通投票插件拉开差距的关键。

## 互动建议
评委在关键节点只能做少量高价值决策，例如：
- “帮我收口这局”
- “匿名发起倡议”
- “发起时间投票”
- “提醒未回复成员”
- “确认成局”
- “生成回忆卡”

## 非目标
- 不追求真实的自然语言理解极限
- 不做大模型能力秀
- 不做复杂管理后台
- 不做长期记忆系统的真实实现
