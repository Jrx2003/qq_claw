# 11 真实 QQ 平台约束说明（P1 参考，不作为 P0 阻塞项）

> 这个文件帮助团队知道：如果后续要把 Demo 变成一个更接近真实接入的原型，需要面对哪些平台事实。
> 但 P0 不应被这些约束绑死。

## 1. 平台支持什么
QQ 机器人官方文档明确说明：
- 机器人可以被添加到群聊 / 频道内互动
- 用户也可以直接与机器人单独对话
- 创建机器人后，会获得 AppID / AppSecret
- openapi 通过 Access Token 鉴权

## 2. 为什么 P0 不建议先真接入
因为真实接入需要处理：
- 鉴权
- 事件订阅
- 消息频控
- 唯一身份映射
- 交互确认
- URL 白名单
- 客户端差异

这些都不是当前评委最关心的价值本体。

## 3. 对题目 3 最重要的真实约束
### 群里最稳的触发方式是 @bot
官方事件里，群聊与机器人最核心的消息事件是 `GROUP_AT_MESSAGE_CREATE`，即：
用户在群聊中 @ 机器人发送消息，机器人收到事件。

这意味着真实版本里不应假设：
- 机器人默认能旁听所有群消息
- 机器人默认能静默感知所有意图

更稳的设计是：
- 用户主动 @ 虾局长
- 或点击消息按钮继续推进

### 按钮交互需要挂在 markdown 消息下
官方文档说明：
- keyboard 按钮依附于 markdown 消息
- 不支持只下发键盘消息
- 用户点击回调按钮会触发 `INTERACTION_CREATE`
- 后端需要对 interaction 进行 ACK，否则客户端会持续 loading

### 身份 ID 不是统一不变的
官方唯一身份机制说明：
- 不同 bot 的 openid 不同
- 群是 `group_openid`
- 同一 bot 下，同一个用户在不同群中的群身份 ID 也不同，叫 `member_openid`
- 单聊是 `user_openid`

所以如果未来要做“群里识别某人 → 单聊继续补全”，需要显式绑定，而不是想当然地直接打通。

## 4. 消息发送的现实限制
官方发送消息文档存在一个很重要的现实判断：
- 文档写明主动推送能力自 2025-04-21 起不再提供，调用会报错
- 同一页文档又保留了历史频控说明

因此更稳妥的工程策略是：
- 真实版本默认不要依赖主动推送
- 以被动回复、按钮交互和用户主动触发为主
- 把“主动提醒”在 P0 中先当作产品展示概念，不依赖真实推送一定成功

## 5. 真接入时的建议栈
如果未来真的接：
- 前端：Next.js / React
- 后端：FastAPI 或 Node
- 状态流：LangGraph / 自定义状态机
- 存储：Postgres / Redis
- 平台适配：自写 QQ adapter，不让 SDK 决定业务结构

## 6. 给 Codex 的边界
P0 只需要：
- 在视觉和交互上模拟 QQ 群聊
- 保留未来接 QQ 的接口抽象
- 不要为了“像真实机器人”牺牲 Demo 完成度

## 7. 后续真接入优先级
1. 群聊 @bot 触发
2. markdown + keyboard
3. interaction ACK
4. 显式绑定单聊身份
5. 再考虑真实 reminder 与 memory
