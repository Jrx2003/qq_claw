# 19 提交材料与仓库内容对齐说明

## 1. 为什么仓库里必须有提交材料目录

这不是一个只给工程师看的 repo。
这个项目的仓库，本质上应该服务三类使用者：

1. 你自己和 Codex：持续迭代
2. 评委：理解产品、访问 demo
3. 你录视频和写 PDF 时：快速找素材

所以仓库内必须内生化比赛材料，而不是到最后手忙脚乱去拼。

## 2. 本仓库应当直接产出的材料

### A. 在线 Demo
放到可公开访问 URL：
- 无 LLM 评审模式
- 如有需要，再有真实 LLM 工作台

### B. 产品方案 PDF 素材
仓库里要有一个 `submission/PDF_OUTLINE.md`
至少涵盖：
- 问题定义
- 用户洞察
- 方案结构
- AI 原生性
- 关键场景
- 演示截图
- 落地路径
- 商业化与扩展想象

### C. 录屏脚本
仓库里要有：
- `submission/VIDEO_SCRIPT_OUTLINE.md`

至少涵盖：
- 开场问题
- 主线演示
- 支线闪示
- 结束总结

### D. 截图与画面清单
仓库里要有：
- `submission/SCREENSHOT_SHOTLIST.md`

至少涵盖：
- 首屏群聊
- 成局卡
- 投票进展卡
- 确认成局卡
- 回忆卡
- 匿名倡议卡
- 冲突桥梁卡
- 游戏局卡 / 游戏回忆卡
- studio 模式的 live LLM 证明图

### E. 评委说明页
仓库里要有：
- `submission/README_FOR_JUDGES.md`

内容包括：
- 这是一个什么项目
- 推荐访问哪条链接
- 最佳体验时长
- 先看什么，后看什么
- 如果要自由探索可点哪里

## 3. 提交材料与项目功能的映射

### 主视频里必须出现
- 周五烤肉局主线
- 虾局长承担组织责任
- AI 生成的回忆卡
- 至少一个支线能力闪示

### PDF 里必须强调
- 高频 × 高痛
- AI 原生性
- 群主职责重构
- 匿名倡议的价值
- 冲突桥梁的差异点
- 腾讯游戏局扩展

### 在线 Demo 必须支持
- 评委自己点一次主线
- 你自己自动录一遍主线
- 你自己切出支线讲 20~30 秒

## 4. 建议的仓库目录

```text
submission/
  README_FOR_JUDGES.md
  PDF_OUTLINE.md
  VIDEO_SCRIPT_OUTLINE.md
  SCREENSHOT_SHOTLIST.md
  MATERIALS_CHECKLIST.md
  exported/
    final-pdf/
    final-video/
    screenshots/
```

## 5. 最终应该能从仓库里直接拎出来的东西

你在最后提交前，不应该再“重新组织一次世界”。
理想状态是：

- PDF 文案主体在仓库里
- 录屏脚本在仓库里
- 所有截图位点在仓库里
- 线上 demo 路由在仓库里
- judges README 在仓库里

这样比赛要求的材料就不是额外负担，而是项目本身的一部分。

## 6. 这份文档对 Codex 的要求

Codex 在改代码时，不能只改 `src/`。
它还必须：
- 补 `submission/`
- 补截图位点说明
- 补评委说明页
- 补对外访问说明

这样项目才算真正准备好去交比赛。
